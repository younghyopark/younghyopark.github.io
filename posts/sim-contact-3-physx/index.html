Contact in 3 Simulators, Part 3 — PhysX

2026-05-07 · Younghyo Park
Series Contact in 3 simulators
Part 1 — Intro · Part 2 — MuJoCo · Part 3 — PhysX · Part 4 — Drake
A code-level tour of how PhysX (NVIDIA) determines and resolves contact. PhysX shares MuJoCo's worldview — contact-as-constraint — but solves the resulting problem with a different machinery: persistent manifolds carried across timesteps and impulse-based iterative solvers (PGS and TGS). All file paths are relative to references/PhysX/.

1. Where to look

PhysX's source tree is laid out by abstraction layer. The relevant directories for contact are:

  • physx/include/ — public headers, including the PxBroadPhase / PxMaterial / PxContact structs the user touches.
  • physx/source/geomutils/ — geometry queries: GJK, EPA, persistent contact manifolds, narrowphase dispatch.
  • physx/source/lowlevel/ and physx/source/lowleveldynamics/ — the solver, contact reduction, material combination, articulation.
  • physx/source/lowlevelaabb/, physx/source/gpubroadphase/ — broadphase backends.

As with MuJoCo, the work splits into broadphase → narrowphase → constraint setup → solve. The shapes of those stages are similar; the implementation choices differ in ways worth noticing.

2. Step 1a — Broadphase: five algorithms, one default

PhysX exposes five broadphase algorithms and lets the user pick at scene-creation time. The enum is here:

physx/include/PxBroadPhase.h

struct PxBroadPhaseType {
  enum Enum {
    eSAP,    // 3-axis sweep-and-prune; good for many sleeping objects
    eMBP,    // multi box pruning; better when objects move (needs world bounds)
    eABP,    // automatic box pruning; hybrid, manages regions automatically
    ePABP,   // parallel ABP; can be fastest, more memory
    eGPU     // GPU SAP variant, ABP-style pair generation
  };
};

The default is eABP, set in physx/source/simulationcontroller/src/ScScene.cpp. The user picks one via PxBroadPhaseDesc::mType (PxBroadPhase.h:229). Compare with MuJoCo's single covariance-aligned sweep-and-prune: PhysX makes broadphase a configuration knob because the right choice depends on whether your scene is a city of mostly-sleeping objects (SAP wins) or a small set of fast-moving ones (MBP/ABP win).

CPU broadphases live behind BpBroadPhase.h (physx/source/lowlevelaabb/include/BpBroadPhase.h); the GPU path is PxgBroadphase.h (physx/source/gpubroadphase/include/PxgBroadphase.h).

3. Step 1b — Narrowphase: function tables, GJK/EPA, and persistent manifolds

Narrowphase dispatch in PhysX is, as in MuJoCo, a function table indexed by geom-type pair:

physx/source/geomutils/src/contact/GuContactMethodImpl.h:120

// Per-pair contact functions. Macro GU_CONTACT_METHOD_ARGS
// expands to a uniform signature.
GU_CONTACT_METHOD(contactSphereSphere);
GU_CONTACT_METHOD(contactConvexConvex);
// ... and dozens more pair-specific entry points

There are two parallel dispatch paths in this header. The plain contact* functions are the legacy direct-GJK path; the pcm* functions (line ~191 onward) go through the Persistent Contact Manifold machinery, which is the main path in modern PhysX.

3.1. GJK and EPA, in-house

PhysX ships its own GJK and EPA, not as external libraries. The entry points are:

physx/source/geomutils/src/gjk/GuGJKTest.h:41

GjkStatus testGjk(...);             // separation test
GjkStatus testGjkPenetration(...);  // penetration test
GjkStatus testEpaPenetration(...);  // EPA: depth + normal under penetration

These return contact-relevant data — the closest features on each shape, the separation distance, and (in the EPA case) the penetration depth and normal — in the same way MuJoCo's engine_collision_gjk.c does. The math is the same. The implementations differ in tuning and SIMD layout but not in algorithmic kind.

3.2. The persistent contact manifold

This is where PhysX diverges sharply from MuJoCo. In MuJoCo, every contact you see this timestep was generated this timestep; the collision routines run from scratch and the mjData.contact array is rebuilt each call. In PhysX, contacts are cached in a persistent contact manifold attached to the pair, and most frames the narrowphase only does cheap incremental work on top of the cache.

physx/source/geomutils/src/pcm/GuPersistentContactManifold.h:144

class PersistentContactManifold {
  // Stores up to GU_MANIFOLD_CACHE_SIZE (=4) contacts
  // across timesteps for this pair.
  PxU32 mNumContacts;
  PersistentContact mContactPoints[GU_MANIFOLD_CACHE_SIZE];

  PxQuat mRelativeQuat;        // pair-relative rotation last frame
  PxTransform mRelativeTransform;  // pair-relative pose last frame

  // invalidate_BoxConvex / invalidate_SphereCapsule etc.
  // decide whether the cached contacts are still good.
};

The cache holds at most 4 contacts — GU_MANIFOLD_CACHE_SIZE is hardcoded to 4 (GuPersistentContactManifold.h:44). How those slots get filled is the more interesting story.

3.3. The per-frame dispatch: refresh, incremental, or full regenerate

Each contact-test call (e.g. pcmContactConvexConvex, physx/source/geomutils/src/pcm/GuPCMContactConvexConvex.cpp:194) walks the same three stages:

physx/source/geomutils/src/pcm/GuPCMContactConvexConvex.cpp:228

// 1. Refresh: project cached contacts to current poses,
//    drop any that have separated or drifted too far.
manifold.refreshContactPoints(aToB, projectBreakingThreshold, contactDist);
const bool bLostContacts = (manifold.mNumContacts != initialContacts);
...
// 2. Run GJK once for the closest features in the current pose.
//    GJK is cheap; this happens every frame.
GjkStatus status = gjkPenetration<...>(...);

// 3. Decide: if pose has changed too much, contacts were lost,
//    or the new GJK normal disagrees with the cached one by > 45°,
//    invoke the expensive full SAT-based regeneration. Otherwise,
//    just append the GJK contact incrementally.
if(bLostContacts || manifold.invalidate_BoxConvex(curRTrans, ...)) {
  // full regenerate (see §4)
} else {
  // incremental: addManifoldPoint with the new GJK contact
}

The invalidation thresholds are explicit numerical constants:

physx/source/geomutils/src/pcm/GuPersistentContactManifold.cpp:173

// Translational threshold: 50% of margin (more lenient slots
// for stable cases, tighter for fast-moving).
const PxF32 invalidateThresholds[5] = { 0.5f, 0.125f, 0.25f, 0.375f, 0.375f };

// Rotational threshold: quat dot product. 0.9998 ≈ ~1.1° rotation.
const PxF32 invalidateQuatThresholds[5] = { 0.9998f, 0.9999f, 0.9999f, 0.9999f, 0.9999f };

In other words: for slowly-evolving contact (a box settling on a table, or a finger pressing into a cube), PhysX runs only the cheap GJK-once-plus-append path each frame. The expensive analytic multi-contact generation runs only when the pair's pose has moved more than ~1° relative or 50% of the contact margin, or when the refresh step lost cached contacts.

Two consequences. First, the narrowphase amortizes well across frames — for resting contact, the same 4-slot manifold can persist for many frames with only a single GJK call per frame feeding it. Second, the cached contacts give the iterative solver something to warm-start on: the impulses from last frame's solve are still meaningful for this frame's, and starting the iteration from those impulses rather than from zero converges much faster. This is the operational reason PhysX feels so responsive in scenes with lots of resting contact.

4. Where the multiple contacts come from: SAT + face clipping

Now the question I glossed over: when the manifold does need a full regenerate, where do those 4 contacts come from? The answer is in generateFullContactManifold (physx/source/geomutils/src/pcm/GuPCMContactGenBoxConvex.cpp:532), which is the analog to MuJoCo's multiCCD — same problem, very different mechanism.

PhysX's approach is the textbook Separating Axis Theorem (SAT) + face clipping pipeline:

  1. Find the minimum-overlap axis by testing all face normals (both shapes) and all edge-edge cross-products. The axis with the smallest interval overlap is the contact normal.
  2. Identify a reference face (the face whose normal won the SAT test) and an incident face (the most-anti-parallel face on the other shape).
  3. Project both faces into 2D, perpendicular to the contact normal. Now you have a reference polygon and an incident polygon in a shared 2D frame.
  4. Clip the incident polygon against the reference polygon (Sutherland-Hodgman style), keeping every surviving vertex that has nonpositive separation in 3D. generatedContacts (GuPCMContactGenBoxConvex.cpp:331) is the function that runs this clip.
  5. Reduce to 4 contacts via PersistentContactManifold::addBatchManifoldContacts using a spread heuristic: the deepest contact, then the contact farthest from it, then the one farthest from the line between them, then the one most-perpendicular to the resulting plane.

The mechanism difference from MuJoCo's multiCCD is worth pausing on. MuJoCo gets multiple contacts by perturbing the geoms by ±0.001 rad and re-running the penetration query, then accepting any new distinct contacts. PhysX gets multiple contacts by analytically clipping the two contact faces against each other in 2D. PhysX's path is exact for polygonal convex shapes and produces the full overlap polygon; MuJoCo's is empirical and probes for whether a patch exists at all. They reach similar 4–5-contact outputs by very different roads.

Drake's hydroelastic, in Part 4, also clips polygons against each other — tetrahedra against an equilibrium plane — but the polygons there are the contact surface mesh, not 2D projections of contact faces. The shapes of the algorithms are closer cousins than they look at the table-comparison level.

4.1. Two reducers, not to be confused

PhysX has two independent contact-reduction stages, each with its own implementation, and it's easy to mix them up:

  • Per-pair manifold reduction (PersistentContactManifold::addBatchManifoldContacts and reduceContactsForPCM, GuPersistentContactManifold.cpp:603). This is the one that caps a single pair's contacts to 4, with the spread heuristic above.
  • Per-actor solver-side reduction (Dy::ContactReduction<6, 6>, physx/source/lowleveldynamics/src/DyContactReduction.h:81, invoked from DyDynamics.cpp:420). When two actors have multiple shape pairs touching (e.g. compound shapes), this second-stage reducer caps the total at most 6 patches × 6 contacts before the solver sees them. Same heuristic family, different scope.

A header comment on the solver-side reducer is candid about the 4-vs-5 tradeoff: "might be OK with 4 but 5 guarantees the deepest + 4 contacts that contribute to largest surface area" (DyContactReduction.h:43). They picked 4 for the PCM cache and 6×6 for the solver-side cap; both are pragmatic choices, not theoretically motivated ones.

5. Anatomy of a PxContactPoint

PhysX's per-contact struct is in the public header:

physx/include/geomutils/PxContactPoint.h:39

struct PxContactPoint {
  PxVec3  normal;             // contact normal (world)
  PxReal  separation;         // negative = penetration
  PxVec3  point;              // contact point (world)
  PxReal  maxImpulse;
  PxVec3  targetVel;          // optional driven velocity
  PxReal  staticFriction;     // already-combined
  PxU8    materialFlags;
  PxU32   internalFaceIndex1; // mesh/heightfield only
  PxReal  dynamicFriction;    // already-combined
  PxReal  restitution;        // already-combined
  PxReal  damping;            // only for compliant contacts
};

This is qualitatively different from MuJoCo's mjContact. Where mjContact carries solver regularization parameters (solref, solimp), PxContactPoint carries combined material coefficients: the static and dynamic friction, the restitution, and (for compliant contacts) damping. The pair's two geom materials have already been combined into single numbers by the time this struct is filled.

At a higher level, contacts are grouped into a PxContactPatch:

physx/include/PxContact.h:56

struct PxContactPatch {
  PxVec3 normal;
  PxReal restitution;          // patch-level (combined)
  PxReal dynamicFriction;
  PxReal staticFriction;
  PxU16  materialIndex0, materialIndex1;
  PxU16  startContactIndex;    // index into contact list
  PxU8   nbContacts;
};

All contacts in a patch share a normal and material; the patch exists so the solver can iterate over a contiguous block with one set of friction parameters. Note again: the combined values are on the patch, with material-index back-references kept around mostly for reporting.

5.1. Material combination

Where do the combined values come from? Each material has a combine mode, and the combination is mechanical:

physx/include/PxMaterial.h:123

struct PxCombineMode {
  enum Enum {
    eAVERAGE = 0,
    eMIN     = 1,
    eMULTIPLY = 2,
    eMAX     = 3
  };
};

At pair time, PhysX picks the mode by taking the maximum of the two materials' combine modes (a documented rule, also at PxMaterial.h:112) and applies it:

physx/source/lowlevel/software/include/PxsMaterialCombiner.h:70

void PxsCombineMaterials(const PxsMaterialData& mat0,
                         const PxsMaterialData& mat1,
                         PxReal& combinedStaticFriction,
                         PxReal& combinedDynamicFriction,
                         PxReal& combinedRestitution,
                         PxU32&  combinedMaterialFlags,
                         PxReal& combinedDamping);

There's a special case for compliant contacts — if exactly one of the two materials uses compliant mode (negative restitution, see §7), the combiner switches to eMIN so the compliant behavior dominates (PxsMaterialCombiner.h:96-98). Otherwise it's a straightforward apply-the-mode-elementwise.

6. The solver: PGS and TGS

PhysX has two interchangeable constraint solvers. Both are iterative impulse-based solvers: they apply velocity-level impulses at each contact, sweep through all constraints, and repeat. They do not solve a global optimization problem the way MuJoCo's Newton solver does.

6.1. PGS — Projected Gauss-Seidel

PGS is the classical iterative solver for this kind of problem, and it's what PhysX has shipped for years.

physx/source/lowleveldynamics/src/DySolverConstraints.cpp:47

// solve1D: process a single 1D constraint.
// Computed impulse:
//   lambda' = [vT - (vn0 + vni) - b] / r
// where r = J M^-1 J^T is the effective inverse mass
// of the constraint, b is the bias term, vT is the target,
// vn0 / vni are linear / "mom-velocity" projections.
// Lambda' is then clamped to [minImpulse, maxImpulse] and
// applied to both bodies.

The solver iterates this loop over all constraints (contacts + joints + friction) for some number of velocity iterations, then over position iterations to clean up residual penetration. The applied impulse is accumulated in SolverConstraint1D::appliedForce (DySolverConstraint1D.h:70), which is what gets warm-started from the manifold cache.

Per-contact friction is handled by adding 1 or 2 extra constraints to the iteration loop:

physx/source/lowleveldynamics/src/DyContactPrep.h:145

createFinalizeSolverContactsCoulomb1D(...);  // 1 friction direction (cheap)
createFinalizeSolverContactsCoulomb2D(...);  // 2 orthogonal friction directions

This is a pyramidal friction approximation, the same way MuJoCo's pyramidal cone option is. PhysX does not have an elliptic / true second-order cone option in the rigid-body solver; it ships pyramidal only. For most scenes (and for games, where PhysX grew up) this is fine. For dexterous manipulation where diagonal friction direction matters, it's an approximation worth knowing about.

6.2. TGS — Temporal Gauss-Seidel

TGS is the newer solver, recommended for "stiff" content (articulated robots, heavy machinery, dense piles).

physx/source/lowleveldynamics/src/DyTGS.h:29

// TGS divides the timestep into substeps and runs
// solver iterations per substep. Each substep advances
// positions, re-evaluates constraint errors, and re-solves.
void integrateCoreStep(...);

The mechanical change from PGS to TGS is that, between solver iterations, TGS partially advances the positions, then re-evaluates the constraint errors against the new positions. PGS doesn't — it iterates at a single position. The effect is that TGS gives the iterative solver something more like an implicit treatment without making each iteration much more expensive: stiff constraints don't blow up the way they do in PGS at the same iteration count.

This is roughly analogous to MuJoCo's implicit-in-velocity treatment of the reference acceleration (Part 2, §6), but it's achieved by a different mechanism — substepping inside the solver rather than co-solving a single optimization problem. The user-visible effect is similar: more stable behavior with stiff contact at the same simulation timestep.

7. Compliant contacts via negative restitution

PhysX has no concept of a constitutive material model in the rigid contact solver, but it has a small back-door for "soft" contact behavior: negative restitution.

physx/include/PxMaterial.h:188

// "If a negative value is provided it is interpreted
//  as stiffness term for an implicit spring."
void setRestitution(PxReal rest);

When the value is negative, PhysX reinterprets it as a spring stiffness; the contact constraint becomes a spring-damper with that stiffness and the material's damping field as the damper. This is the same family of trick MuJoCo uses with solref: a regularized constraint dressed up as a spring. It's not a constitutive model — there's no Young's modulus, no contact area, no integration over a region — but it lets users get bouncy/squishy behavior without leaving the rigid solver.

8. Articulation: reduced-coordinate Featherstone

Articulated chains (robot arms, ragdolls) get a reduced-coordinate treatment via Featherstone:

physx/source/lowleveldynamics/include/DyFeatherstoneArticulation.h:29

class FeatherstoneArticulation {
  // Joint-space dynamics; no Cartesian-space integration of
  // each link as a free body. Joint constraints are implicit
  // in the parameterization.
};

Contacts on articulated links are wrapped into solver constraints through a separate prep path (DyArticulationContactPrep.h), so the same PGS/TGS iteration loop still applies; only the bodies' degree-of-freedom layout differs. PhysX also retains a maximal-coordinate articulation path for backward compatibility, but the modern recommendation is reduced-coordinate.

9. Recap: a single timestep, narrated

  1. Broadphase (chosen from SAP/MBP/ABP/PABP/GPU, default ABP) drops obvious non-colliding pairs.
  2. Refresh cached contacts: project last frame's manifold contacts to the current poses, drop any that have separated past the breaking threshold.
  3. Run GJK once for the deepest closest-feature pair under the current pose. Cheap; runs every frame on every surviving pair.
  4. Decide: incremental or full? If contacts were lost in refresh, or the relative pose moved past the invalidation thresholds (~50% of margin in translation, ~1° in rotation), or the GJK normal disagrees with the cached manifold normal by > 45°, run the full SAT + face-clipping regenerator (generateFullContactManifold). Otherwise append the GJK contact incrementally via addManifoldPoint, displacing one of the 4 cached slots if needed via the spread heuristic.
  5. Per-pair reduction: when the full path runs, addBatchManifoldContacts caps the manifold at 4 contacts using the deepest+farthest+farthest-from-segment heuristic.
  6. Material combination turns the two materials' friction/restitution/damping into single per-contact (and per-patch) values via the chosen combine mode.
  7. Constraint prep: each contact becomes one normal constraint plus 1 or 2 pyramidal friction constraints, with impulse bounds set from the combined material values.
  8. Solve: PGS or TGS iterates impulses over all constraints, warm-started from last frame's appliedForce values.
  9. Velocities and positions integrate; manifolds save their contacts and impulses for the next step.

The thing to keep in mind, going into Part 4, is that PhysX is squarely in the same family as MuJoCo — contact-as-constraint, force is the solver's output — with the implementation choice of persistent state plus iterative impulses instead of MuJoCo's fresh state plus convex optimization. Drake answers a different question entirely.

10. A short reading list

  • NVIDIA's PhysX SDK documentation, particularly the chapters on rigid-body dynamics, friction, and articulations. The TGS solver section is the clearest written description of how the substepping changes the math.
  • Catto, "Iterative Dynamics with Temporal Coherence" (GDC 2005). The textbook reference for the PGS-on-impulses formulation that PhysX (and Box2D, and Bullet) descends from.
  • DySolverConstraints.cpp, DyContactReduction.h, GuPersistentContactManifold.h. The three files together cover most of what's distinctive about PhysX's pipeline.
