A code-level tour of how MuJoCo turns geometry into contact forces — broadphase, narrowphase, solref/solimp, and the convex solver. Read Part 1 first if you want the framing for why I'm comparing three engines at all; this part is purely a walkthrough of the MuJoCo source.

1. Where to look

MuJoCo's contact code lives in src/engine/ in the google-deepmind/mujoco repo. There are about a dozen files involved, but the work splits cleanly into three stages, in order:

  1. Find contacts. Figure out which geoms are touching or about to touch, and where.
  2. Build the constraint problem. Turn that list of contacts into a numerical optimization problem in which the contact forces are the unknowns.
  3. Solve it. Use a convex solver (Newton, CG, or PGS) to find contact forces and the resulting accelerations self-consistently.

We'll walk each stage. The comments in the source are good. The problem is that the steps are spread across enough files that nobody assembles them for you in one place — so that's what this post does.

2. Step 1a — Broadphase: which pairs even need looking at?

A scene with \(N\) geoms has \(N(N-1)/2\) possible pairs. Almost none of them are touching. The first job of collision detection is to throw out the obvious non-collisions cheaply, before doing any expensive geometry work.

MuJoCo uses a covariance-aligned sweep-and-prune algorithm. The trick: compute the principal-axis frame of all the geoms (an eigendecomposition of their position covariance matrix), project every geom's bounding sphere onto that frame's longest axis, sort by projected position, and only check pairs whose projections overlap. This is a one-shot rotation per timestep that aligns the sweep direction with the scene's longest spread — tall stacks get swept vertically, wide tables get swept horizontally — and it makes the linear sweep much more discriminative than a fixed world-axis sweep would be.

You can see it being computed here:

src/engine/engine_collision_driver.c

// construct covariance-aligned 3D frame
mju_eig3(eigval, frame, quat, cov);

The output of broadphase is a flat array of "this pair survived, go check it carefully" — a few thousand candidate pairs in a typical robot scene, down from the naive quadratic count.

3. Step 1b — Narrowphase: where are the contact points?

For each surviving pair, MuJoCo dispatches to a type-specific collision routine. There's a 2D function table indexed by (geom_type_a, geom_type_b) — e.g. (sphere, plane), (capsule, box), (box, box). For primitive pairs, there's a hand-written closed-form routine in engine_collision_primitive.c that returns the contact point, normal, and penetration depth analytically. For non-primitive pairs — meshes, convex hulls, anything not on the closed-form list — the dispatch falls through to the general-purpose GJK + EPA pipeline in engine_collision_gjk.c.

GJK (Gilbert-Johnson-Keerthi) is the standard textbook algorithm for testing whether two convex shapes intersect; EPA (Expanding Polytope Algorithm) is the followup that, given that they do intersect, finds the contact normal and penetration depth. Together they handle the long tail of geometry that doesn't fit a primitive case. There's also a separate path for SDF-based collisions in engine_collision_sdf.c for geoms that have a signed-distance function defined.

Three details are worth surfacing here, because they show up later in comparisons with the other engines:

(a) Multi-contact for some primitive pairs. Most primitive narrowphase routines return one contact point per pair. The exception is box-box, in engine_collision_box.c, which can return up to 8 contact points for a single pair. That's because a box resting flat on another box doesn't have a unique "contact point" — it has a face. If you returned only one point, the resting box would teeter and drift. So the box-box collider explicitly enumerates the corners of the overlapping region and emits a contact at each, giving the solver enough information to pin all the rotational degrees of freedom.

(b) multiCCD: multi-contact for general convex pairs. What about a cylinder resting on a box, or two arbitrary meshes face-to-face? Box-box has hand-coded enumeration, but for general convex-convex pairs through the libccd / GJK / EPA path you'd naively get only the single deepest contact — same teeter-and-drift problem. MuJoCo solves this with multiCCD, controlled by the multiccd flag (on by default; the disable bit is mjDSBL_MULTICCD). The trick is in engine_collision_convex.c:840:

src/engine/engine_collision_convex.c:860

// look for additional contacts
if (ncon == 1 && !mjDISABLED(mjDSBL_MULTICCD)
    && m->geom_type[g1] != mjGEOM_ELLIPSOID && m->geom_type[g1] != mjGEOM_SPHERE
    && m->geom_type[g2] != mjGEOM_ELLIPSOID && m->geom_type[g2] != mjGEOM_SPHERE) {

  // multiCCD parameters
  const mjtNum relative_tolerance = 1e-3;
  const mjtNum perturbation_angle = 1e-3;

The algorithm: after the initial GJK/EPA call returns one contact, pick two axes in the contact's tangent plane and four (axis, sign) combinations of a tiny ±0.001-radian rotation. For each, rotate geom 1 by that angle around the contact point and counter-rotate geom 2; re-run the penetration query; if a new distinct contact is found (more than 1e-3 × min(rbound) from existing ones), keep it. Restore the original poses and move on. Up to four extra contacts can be added this way, for a total of five.

The insight is that for a face-on-face contact, tilting the geoms by a hair reveals contact points elsewhere on the face — the shape is locally flat, so a small rotation moves the deepest point. For a true point contact (sphere, ellipsoid), the perturbation produces no distinct new contact, so the algorithm stays at one. It's an empirical "is this a point or a patch?" probe rather than an analytic enumeration. The cap of 4–5 contacts per pair for mesh-mesh and mesh-box (with or without margin) is set in maxContacts:

src/engine/engine_collision_driver.c:114

int is_multiccd = !mjDISABLED(mjDSBL_MULTICCD);
if (!is_multiccd) {
  return 1;  // fall back to single contact
}

if (type1 == mjGEOM_CAPSULE || type2 == mjGEOM_CAPSULE ||
    type1 == mjGEOM_CYLINDER || type2 == mjGEOM_CYLINDER) {
  return 5;
}
// 4 contacts for mesh-mesh or mesh-box without margins, 5 with margins
return has_margin ? 5 : 4;

This changes the practical picture significantly: MuJoCo's multi-contact story isn't "box-box only, one-per-pair otherwise" but "box-box gets 8, capsule/cylinder/mesh pairs get up to 5 via multiCCD, true point contacts (sphere, ellipsoid) get 1." It's still not a continuous contact patch the way Drake's hydroelastic surface is — the points are sampled, not integrated — but it's closer to a distributed force than the "one point per pair" reading of MuJoCo would suggest.

(c) Contacts are events, not state. Every contact MuJoCo finds in this step is added to the mjData.contact array via mj_addContact. If two bodies are separating — about to lose contact — the contact simply isn't added on the next timestep. There's no smooth birth/death of contact patches, and no carry-over of contact state across timesteps. Contacts appear when the narrowphase reports penetration (or proximity, given a margin) and disappear when it doesn't. PhysX takes the opposite approach (persistent manifolds), which is why its contact behavior feels qualitatively different even on identical geometry. We'll see that in Part 3.

4. Anatomy of an mjContact

Once the narrowphase has finished, we have a list of contacts. Let's pause and look at what one of them actually contains, because this struct is where MuJoCo's whole philosophy gets baked in:

include/mujoco/mjdata.h

struct mjContact_ {
  mjtNum  dist;                    // distance; negative = penetration
  mjtNum  pos[3];                  // contact midpoint
  mjtNum  frame[9];                // normal [0-2], tangents [3-8]

  mjtNum  friction[5];             // [0-1] = mu, [2] = spin, [3-4] = roll
  mjtNum  solref[mjNREF];          // solver reference (normal)
  mjtNum  solreffriction[mjNREF];  // solver reference (friction)
  mjtNum  solimp[mjNIMP];          // solver impedance

  int     dim;                     // 1, 3, 4, or 6
  int     geom[2], flex[2], elem[2], vert[2];
};

Read this carefully. There is no force in this struct. There is no pressure, no contact area, no Young's modulus, no material property. An mjContact is a geometric object, not a physical one: it tells the solver "two bodies are this far into each other at this point, here's the local frame, here's how squishy it should feel, here's the friction coefficient." Whatever force comes out of the contact later is the solver's problem.

The fact that the contact carries regularization parameters rather than material parameters is the heart of the constraint view of contact. Drake's ContactSurface looks completely different (it carries a pressure field); we'll see that in Part 4.

5. Step 2 — solref and solimp: the soft-contact spring

Now the work of turning these geometric contacts into a numerical optimization problem. This step lives in engine_core_constraint.c.

MuJoCo's "soft contacts" are regularized — meaning instead of a hard non-penetration constraint (distance ≥ 0), the solver enforces a softened version, modeled as a virtual spring-damper that pushes bodies apart in proportion to how deeply they've penetrated, with some damping to take energy out. The two parameter families solref and solimp together control exactly what this virtual spring-damper looks like. Once you know what they're parameterizing, the names start making sense.

The five-second version is:

  • solref controls the spring-damper reference motion. In the normal positive-valued mode, you don't enter stiffness and damping directly. You enter a time constant and a damping ratio, and MuJoCo converts those into stiffness-like and damping-like coefficients in constraint space. Smaller time constant means faster correction. Lower damping ratio means more bounce or ringing.
  • solimp controls the constraint impedance d(r). This is MuJoCo's "how much authority does this constraint have right now?" knob. Small impedance means a weak constraint; large impedance means a strong one. Because solimp can make d depend on penetration, contact can start weak near first touch and harden as bodies sink in.

So a useful mental shortcut is:

  • solref: what spring-damper motion the constraint wants.
  • solimp: how strongly the solver enforces that motion at this penetration depth.

In that sense, the right intuition is a penetration-dependent spring-damper. But the precise object in the docs is not a material spring with stiffness in N/m. It is a soft constraint with three internal quantities: impedance d, damping b, and stiffness k. The reference acceleration is spring-damper-like, \(a_\mathrm{ref} = -b v - k r\), and the impedance d controls how much the solver follows that reference motion instead of the unconstrained motion. So solimp is better understood as penetration-dependent constraint authority, not just a literal stiffness multiplier.

That word "authority" is not just a metaphor. MuJoCo turns impedance into the diagonal regularizer used by the solver:

\[ R_{ii} = \frac{1 - d_i}{d_i}\,\widehat{A}_{ii} \]

So when d is close to 1, R is close to 0 and the constraint is almost hard: the solver strongly follows the spring-damper reference acceleration. When d is small, R becomes large and the constraint is weak: the solver mostly allows the unconstrained acceleration. In the ideal diagonal case, the contact acceleration is approximately an interpolation:

\[ a_\mathrm{constrained} \approx d\,a_\mathrm{reference} + (1-d)\,a_\mathrm{unconstrained} \]

This is the clearest way to read solimp: it is a penetration-dependent weight on how much the contact correction is allowed to matter inside the optimization problem.

solref: time constant, damping ratio

solref is two numbers. In its standard mode (the positive-valued case), they describe a damped second-order system:

  • solref[0] = time constant (seconds): how quickly does the spring restore equilibrium?
  • solref[1] = damping ratio (dimensionless, typically 1 = critically damped): how much does it oscillate?

The defaults are solref = (0.02, 1) — a 20 ms time constant, critically damped. The reason these are the parameters (rather than a stiffness k and damping b) is that they're scale-invariant in the right way: a 20-ms-and-critically-damped contact looks the same whether the body weighs a kilogram or a ton, because the solver internally converts to k and b values that account for the body's effective inertia in the constraint direction. From the code:

\[ \begin{aligned} k &= \frac{d(r)}{d_\mathrm{width}^{2}\,\mathrm{timeconst}^{2}\,\mathrm{dampratio}^{2}} \\ b &= \frac{2}{d_\mathrm{width}\,\mathrm{timeconst}} \end{aligned} \qquad \text{(standard mode, } \mathrm{solref}[0] > 0\text{)} \]

where d_width is one of the solimp numbers (we'll get there), and d(r) is the impedance at the current constraint violation. If you set solref[0] negative, you've opted into "direct mode" and you're specifying raw stiffness and damping values explicitly. The positive format is the easier one for ordinary modeling; the direct format is useful when you really do want to identify stiffness and damping-like numbers.

solimp: the impedance shape

solimp is five numbers, and on first read they look like alphabet soup. The function being parameterized is the impedance function d(r): given how large the constraint violation is (for contact, how far into the contact layer we are), how strongly should the constraint be enforced? d ranges from 0 (basically not enforced, squishy) to 1 (basically rigid). The five numbers:

  • solimp[0] = d_0 — impedance at zero penetration (just touching). Default 0.9.
  • solimp[1] = d_width — impedance once the violation reaches width. Default 0.95.
  • solimp[2] = width — how wide a band of penetration depth does the transition occupy. Default 0.001 m.
  • solimp[3] = midpoint — where in [0,1] of that band the transition is centered. Default 0.5.
  • solimp[4] = power — how steeply d rises through the transition (a piecewise polynomial). Default 2 (quadratic).

The function lives at engine_core_constraint.c:2014 in getimpedance. The big idea is "soften when barely touching, harden as you penetrate more," and the five parameters describe a smooth sigmoid-like function that interpolates between d_0 and d_width across a band of width width.

Here's what's important to absorb: none of this is a material model. Nowhere are you specifying that the contact is silicone or steel. solref is a numerical conditioner for the solver — "make the spring stiff enough that bodies don't visibly interpenetrate, soft enough that the optimizer doesn't blow up." solimp shapes how aggressively the regularization kicks in as penetration grows. These are solver hyperparameters, dressed up as physical ones.

This is also why getting MuJoCo to "feel like silicone" is fiddly. There's no E-modulus knob. You back-solve solref and solimp until the steady-state contact force matches what you'd expect from a Young's modulus calculation, but the transient compliance — how the material vibrates and dissipates — can still be wrong, and the engine doesn't care. It's a convex optimizer, not a constitutive law.

6. Step 2 (continued) — The reference acceleration

Once we've turned the spring-damper parameters into \(k\) and \(b\) values, MuJoCo computes a reference acceleration for each constraint:

\[ a_\mathrm{ref} = -b\,v - k\,r \]

where \(v\) is the velocity of the bodies in the contact's normal direction and \(r\) is the position residual (penetration depth, or distance below margin). This is the acceleration the constraint wants — "I want to push the bodies apart at this rate to undo their penetration in the next timestep." The friction constraints get \(a_\mathrm{ref}\) with \(k=0\), so they're pure velocity damping ("don't slide tangentially").

Acceleration of what?

This is the part that is easy to misread. \(a_\mathrm{ref}\) is not the acceleration of either object in the contact. It is the desired acceleration of the constraint error. For a normal contact, that means the desired acceleration of the signed gap between the two contact points.

MuJoCo solves for generalized acceleration: joint accelerations for articulated bodies, and linear/angular accelerations for free bodies. Call that unknown \(x\). A contact constraint then uses its Jacobian \(J\) to map those body/joint accelerations into constraint space:

\[ a_\mathrm{constraint} \approx Jx \]

For one normal contact between bodies A and B, this is roughly the relative acceleration of the two contact points along the contact normal:

\[ a_\mathrm{constraint} \approx n^\mathsf{T}(a_A - a_B) \]

In words: object space says "the top box accelerates upward, the bottom box accelerates slightly downward, this joint accelerates by so much." Constraint space says "the contact gap is accelerating open or closed." The reference acceleration lives in this second space. It is the contact's request:

\[ Jx \approx a_\mathrm{ref} \]

The solver then chooses the actual generalized acceleration \(x\) that best satisfies all those contact requests while also respecting mass, inertia, joints, friction cones, actuators, gravity, and every other constraint in the scene.

This is why the force does not appear first. In the primal view, MuJoCo asks "what acceleration is closest to free motion while satisfying the constraints?" The contact force is the Lagrange multiplier that makes that acceleration happen. So the primal variable is acceleration; the dual variable is force.

Notice this is implicit-in-velocity: the v term is evaluated at the new (post-step) velocity, not the old one. That matters. It means the constraint's damping term is co-solved with the dynamics; it's not an explicit force you slap on top after the fact. This is the secret behind MuJoCo's ability to run stable contact at large timesteps with stiff parameters. PhysX's TGS solver achieves something similar through substepping; Drake's SAP achieves it by formulating the discrete dynamics convexly. We'll see both.

7. Step 3 — The convex optimization

All the contact information is now pre-baked. The actual solve happens in engine_solver.c, and it's the most intellectually elegant part of MuJoCo. The problem is stated as Gauss's principle of least constraint: among all motions consistent with the constraints, the actual motion is the one that minimizes the deviation from the unconstrained motion (weighted by inertia).

There are three accelerations to keep separate:

  • Free acceleration: what the system would do with no contacts or other constraints.
  • Reference acceleration: what each contact wants in constraint space, computed from \(a_\mathrm{ref} = -b\,v-k\,r\).
  • Actual acceleration: what MuJoCo chooses after satisfying all contacts, joints, friction cones, masses, actuators, and gravity together.

The mental model is:

  1. Free motion: what would happen with no constraints?
  2. Contact requests: each contact computes \(a_\mathrm{ref}\).
  3. Global solve: pick one generalized acceleration \(x\) for the whole system.
  4. Dual recovery: the contact forces are the Lagrange multipliers required to enforce that chosen acceleration.

So the optimization is not "compute one force per contact in isolation." It is "choose one physically consistent acceleration for the entire scene."

Concretely, the primal form is something like:

\[ \begin{aligned} \operatorname*{minimize}_{x,\,y}\quad & \frac{1}{2}\left\lVert x - M^{-1}(\tau-c)\right\rVert_M^2 + \text{regularizer}(y) \\[4pt] \text{subject to}\quad & J_E x_E - y_E = 0 \quad\text{(equality)} \\ & J_F x_F - y_F = 0 \quad\text{(friction loss)} \\ & J_C x_C - y_C \in \mathcal{K}^{*} \quad\text{(contact, dual cone)} \end{aligned} \]

where \(x\) is the unknown acceleration, \(M\) is the inertia matrix, \(J\) is the constraint Jacobian, and \(\mathcal{K}^{*}\) is the dual of the friction cone. We're saying "find the acceleration closest to the free-flight one (weighted by mass), subject to not violating any contacts and staying inside the friction cone on each contact." The Lagrangian dual of this primal problem is a smaller optimization over the contact forces themselves — that's the actual thing the code minimizes.

Three solver flavors live in engine_solver.c:

  • Newton (the default): assembles the full Hessian, factorizes with Cholesky, line-searches on the primal cost. Handles both pyramidal and elliptic friction cones. Fastest for moderate problem sizes.
  • CG (conjugate gradient): nonlinear CG with line search; no setup cost; useful for large or rapidly changing constraint sets.
  • PGS (Projected Gauss-Seidel): block-coordinate descent in dual space — the classical algorithm in this area, included for compatibility and as a cheap warm-start. For elliptic cones, PGS solves a 5D quadratically constrained subproblem at each contact.

The cone choice is independent of the solver: 

ctx->is_elliptic = (m->opt.cone == mjCONE_ELLIPTIC);

Pyramidal cones approximate the friction cone with edges (the classical LCP form). The dual constraints become box constraints — cheap to solve. Elliptic cones use the true second-order cone — physically correct (pyramidal cones over-allow friction along the diagonals), but the dual constraints become second-order, slower per iteration. Elliptic is what you want for serious manipulation; pyramidal is the default because it's faster and "good enough" most of the time.

8. Constraint islands and parallelism

One last optimization worth knowing about: not every contact affects every other contact. If your scene has a robot in one corner picking up a cup, and another robot across the room shaking hands with itself, those two contact problems can be solved independently — they share no bodies and no constraints. MuJoCo identifies these constraint islands via a flood-fill on the constraint Jacobian's sparsity pattern (engine_island.c) and farms them out to separate threads. The island decomposition is a small thing, but it's the difference between "real-time on a laptop" and "real-time only on a server" in scenes with many independent agents.

9. Recap: a single timestep, narrated

Pulling it all together, here's what one MuJoCo timestep does to contact, with the names we now have:

  1. Broadphase sweeps geom AABBs along a covariance-aligned axis, drops obvious non-collisions.
  2. Narrowphase dispatches each surviving pair to a primitive routine or GJK/EPA, generating mjContact records with geometric info but no force.
  3. Constraint setup turns solref/solimp/friction into a reference acceleration and an impedance for each contact — the "what the spring-damper wants" part.
  4. Island decomposition splits the contact graph into independent subproblems.
  5. Convex solve, per island, finds contact forces that satisfy Gauss's principle subject to the friction-cone constraint.
  6. Forces map back to accelerations; integrate; repeat.

The thing to keep in mind, going into the next two parts, is that everywhere in this pipeline, "contact" is a discrete geometric event, and "force" is a Lagrange multiplier the solver computes. Nothing in MuJoCo's contact code looks at a material's Young's modulus and computes a pressure. Material-like behavior emerges from the choice of solref/solimp, which are solver hyperparameters that happen to look like a spring-damper.

10. Looking ahead: how PhysX is different

PhysX, in Part 3, mostly asks the same kind of question as MuJoCo: contact is still a constraint problem. The difference is the engineering posture. MuJoCo rebuilds contacts every step, forms a clean convex problem, and solves for a globally consistent acceleration/force answer. PhysX leans harder into time coherence: contacts usually do not change completely from one frame to the next, so it remembers useful contact points, refreshes them, and iteratively repairs motion with impulses.

Question MuJoCo PhysX
Contact lifetime Contacts are rebuilt each step. Contacts live in a persistent manifold and are refreshed.
Multi-contact Special primitive routines plus multiCCD for extra points. Cached manifold points; full regeneration via SAT + face clipping when needed.
Solver style Convex optimization per island; Newton / CG / PGS. Iterative impulse solver; PGS or TGS.
Stability trick Implicit soft constraints and convex regularization. Warm starting, persistent manifolds, and TGS substepping.
Design center Robotics, control, contact-rich RL. Games, Isaac Sim / Isaac Lab, large real-time scenes.

The short version: MuJoCo is elegant because it formulates contact as a clean optimization problem. PhysX is powerful because it exploits time coherence. It remembers the contacts it already found, uses them to warm-start the next solve, and iterates local impulses until the scene is good enough for real-time simulation. Same constraint family; different engineering lineage.
