Contact in 3 Simulators, Part 4 — Drake

2026-05-07 · Younghyo Park
Series Contact in 3 simulators
Part 1 — Intro · Part 2 — MuJoCo · Part 3 — PhysX · Part 4 — Drake
A code-level tour of how Drake (RobotLocomotion) determines and resolves contact. Drake answers a different question than MuJoCo or PhysX: it treats contact as a constitutive boundary condition, computes a contact surface and a pressure field on that surface, and feeds the result to a convex solver. All file paths are relative to references/drake/.

1. Where to look

Drake's most important architectural commitment is that geometry and dynamics are separate components. MuJoCo's contact code lives intermingled with its dynamics; PhysX's is organized in layers under a single SDK. Drake splits them into two systems that talk over a port:

  • SceneGraph + ProximityEngine — everything geometry-related: shape registration, broad and narrow phase, signed distance, and (the part this post is mostly about) hydroelastic contact surfaces. Lives in geometry/.
  • MultibodyPlant — the rigid-body dynamics: mass matrices, joint coordinates, contact-force integration, and time stepping. Lives in multibody/.
  • contact_solvers/sap/ — the Semi-Analytic Primal solver that takes the geometric contact data and returns velocities. Lives in multibody/contact_solvers/.

Within those, the relevant files for contact are concentrated in geometry/proximity/, multibody/plant/, and multibody/contact_solvers/sap/.

2. Two contact models, picked by the user

Unlike MuJoCo and PhysX, Drake offers two distinct contact models on the same plant, selectable at setup time. There's also a third enum value that combines them:

multibody/plant/multibody_plant.h:138

enum class ContactModel {
  kHydroelastic,            // hydroelastic on every pair; throws if any pair is unsupported
  kPoint,                   // classical point contact via FCL on every pair
  kHydroelasticWithFallback // hydroelastic where supported, point contact for the rest
};

The default is not the strict hydroelastic mode. It's the permissive one:

multibody/plant/multibody_plant.h:6638

ContactModel contact_model_{ContactModel::kHydroelasticWithFallback};

This matters in practice. A freshly-constructed MultibodyPlant is in kHydroelasticWithFallback mode, but hydroelastic only actually engages on a geom pair where both geoms have hydroelastic proximity properties registered (we'll see how to do that in §4.1). If you load a URDF with no hydroelastic decoration, every pair falls back to point contact silently. The "with fallback" mode is essentially "try hydroelastic, but don't make this an error if you didn't set it up." The strict kHydroelastic mode is the one that throws an exception on an unsupported pair, and most production users avoid it for that reason. The user-facing way to flip the model is:

multibody/plant/multibody_plant.cc:871

void MultibodyPlant<T>::set_contact_model(ContactModel model) {
  ...
  contact_model_ = model;
}

At simulation time, the plant dispatches on this enum to issue one of three different geometry queries:

multibody/plant/multibody_plant.cc:2992

switch (contact_model_) {
  case ContactModel::kPoint:
    query_object.ComputePointPairPenetration(...);
    break;
  case ContactModel::kHydroelastic:
    query_object.ComputeContactSurfaces(...);
    break;
  case ContactModel::kHydroelasticWithFallback:
    query_object.ComputeContactSurfacesWithFallback(...);
    break;
}

Two different geometry queries, two different output types. The fallback mode runs the hydroelastic query and, for any pair that came back unsupported, runs the point-contact query as a backup. The output of the day is a heterogeneous mix of contact surfaces and point pairs, which the rest of the pipeline handles by branching on type at the integration step (§5).

We'll walk both paths. The point-contact path is short; the hydroelastic path is where most of the interesting code is.

3. The point-contact path

When the user picks kPoint (or when the fallback path hits an unsupported pair), Drake computes contact the way most rigid-body simulators do: penetration as a point pair. Underneath, it leans on FCL (the Flexible Collision Library, an open-source collision-detection library for general convex shapes), which Drake wraps with a Drake-specific callback that packages FCL's output into Drake's data types.

geometry/proximity_engine.h:277

// Returns one PenetrationAsPointPair per intersecting
// geometry pair, computed via FCL.
std::vector<PenetrationAsPointPair<T>>
ComputePointPairPenetration(...) const;

The wrapping is just enough plumbing to ask FCL for one penetration per pair. The callback machinery requests one contact, packages the result, and discards anything beyond:

geometry/proximity/penetration_as_point_pair_callback.h:28

struct CallbackData {
  fcl::CollisionRequestd request;  // enable_contact = true, num_max_contacts = 1
  ...
};
// MaybeMakePointPair extracts FCL's contact and produces
// the Drake-side PenetrationAsPointPair.

The result type is small and familiar:

geometry/query_results/penetration_as_point_pair.h:24

struct PenetrationAsPointPair {
  GeometryId id_A, id_B;     // the two geoms
  Vector3<T> p_WCa, p_WCb;   // closest points (world)
  Vector3<T> nhat_BA_W;      // unit normal: B -> A
  T depth;                   // penetration depth
};

This is essentially the same information MuJoCo's mjContact carries (geometry, no force) and what PhysX's PxContactPoint carries (plus material combination — which Drake puts elsewhere; we'll see in §6). The struct is striking for what's missing: no solref equivalent, no maxImpulse, no combined material values. It's purely geometric, even more so than mjContact.

Three implications worth flagging before we move on:

  • Drake's point contact is one contact per pair, period. There's no analog to MuJoCo's multiCCD or PhysX's SAT face-clipping in this path. Box-on-box contact through kPoint teeters and drifts the same way it would in naive MuJoCo. Drake's answer to "I want distributed contact" is "use hydroelastic," not "make point contact better."
  • FCL is a real dependency. If you've heard of Bullet's collision library or libccd, FCL is in that family — an open-source collision library that handles GJK/EPA, AABB tree culling, and shape-shape primitive routines for the standard types. Drake doesn't reimplement it, just wires its own collision objects through it. So the actual GJK happening on a point-contact pair lives in FCL's source, not Drake's.
  • Hydroelastic doesn't go through FCL. The hydroelastic path has its own broadphase (a BVH on tet meshes) and its own narrowphase (tet-tet polygon clipping in field_intersection.h); it talks to FCL only for bounding-box culling, not for actual penetration queries. We'll see the hydroelastic broadphase in §4.2.

With the point path out of the way, the rest of this post is the hydroelastic path — the part of Drake that's genuinely unlike anything in Parts 2 or 3.

4. The hydroelastic path

When the user picks kHydroelastic, the geometry query is no longer "give me one point per pair"; it's "give me the contact surface between this pair of geoms, with a pressure field defined on it." This requires ahead-of-time work to decorate each geom with a pressure field, then a runtime computation that finds the surface where the two fields balance.

4.1. Pressure fields: the constitutive part

Hydroelastic doesn't ask the user to tune solver parameters. It asks them to specify a material:

geometry/proximity_properties.h:141

// Mark a geometry as compliant. The user provides:
//   hydroelastic_modulus — E (Pa), the stiffness
//   resolution_hint      — how finely to mesh the volume
void AddCompliantHydroelasticProperties(...);

// Mark a geometry as rigid. No modulus needed; treated
// as infinite stiffness.
void AddRigidHydroelasticProperties(...);

// Friction and Hunt-Crossley dissipation, also material-level.
void AddContactMaterial(...); // proximity_properties.h:114

Two things to understand here. First, compliant and rigid geoms are represented differently in memory. Compliant geoms get a tetrahedral volume mesh covering their interior plus a scalar pressure field defined at every vertex. Rigid geoms get a surface triangle mesh and no field at all. The asymmetry exists because rigid is mathematically a degenerate case of compliant — "infinite modulus" — that doesn't need a volume field; you only need to know where the surface is, because any contact surface with a rigid geom must lie on that surface.

Second, the meshing happens once, at registration time. The resolution_hint parameter (in meters) controls how finely each geom is tessellated; smaller values produce more tetrahedra, which yields a smoother contact surface but more work per timestep. Each primitive shape has its own tessellator under geometry/proximity/ (make_box_mesh.h, make_sphere_mesh.h, etc.); the user only sees the resolution hint. After registration, the meshes are fixed; only the pressure field's spatial location changes as the geom moves.

The pressure field on a compliant geom is the actual constitutive law, and it's deliberately simple:

geometry/proximity/make_box_field.h:39

// p(x) = E * e(x), where
//   e(x) = (d(x) - delta) / (H - delta)
//   d(x) = distance from x to the nearest face
//   delta = margin, H = half-size
// Linear extent-to-pressure, gradient magnitude E / (H - delta).
VolumeMeshFieldLinear<T, T> MakeBoxPressureField(...);

The function e(x) is called the extent: 0 at the surface, 1 at the geom's "deepest" interior, with linear interpolation in between. The pressure is just E times that. So a 1 MPa box of half-size 10 cm has a pressure of 0 at its surface and 1 MPa at its center, with the field rising linearly along the gradient toward the center.

It's worth being explicit about what this isn't. e(x) is not strain in the continuum-mechanics sense. It's not the result of solving any deformation equation. It's a geometric proxy — how far you are from the boundary — multiplied by a scalar to give it units of pressure. In the hydroelastic paper this is justified as the small-deformation limit of an elastic body's response, but the model doesn't track actual deformation; the geom remains rigid in its kinematics, and the "pressure" only matters where two such fields overlap.

Still, this simplification is what makes hydroelastic computationally cheap: linear fields on tetrahedra means the surface-where-they-balance computation is closed-form (we'll see in §4.2). A real continuum-mechanics solve would need an FEM iteration per contact and would be orders of magnitude slower. Hydroelastic trades fidelity for tractability and lands somewhere meaningfully more faithful than a single-stiffness spring contact, without actually solving elasticity.

There are analogous field builders for sphere, capsule, cylinder, and convex meshes (make_sphere_field.h, make_capsule_field.h, etc.). The geometry-specific details of "what's the deepest point?" vary by shape:

  • Sphere: deepest point is the center; field is E·(1 − r/R) along the radius.
  • Box: deepest point is the center; field depends on distance to the nearest of the 6 faces.
  • Capsule: deepest is the central spine; isosurfaces are nested capsule shells.
  • Convex mesh: deepest point is computed automatically (medial-axis-like construction during meshing).

Rigid geoms, by contrast, don't get a field at all. The contact surface between a rigid and a compliant geom lives on the surface of the rigid geom wherever the compliant geom's pressure is positive there. Mathematically this is the same equilibrium-surface idea (where the two pressures balance), but with one of the pressures being "infinite outside, zero inside" by convention — collapsing the equilibrium to the rigid surface. This is the rigid-soft case and is its own dispatch (hydroelastic_calculator.h:22), but it shares most of its machinery with the soft-soft case.

This is the moment to flag the philosophical break with Parts 2 and 3. Where MuJoCo's solref and PhysX's stiffness parameters are solver hyperparameters (regularizers that happen to look spring-shaped), hydroelastic_modulus is a material property: a Young's-modulus-like number with units of pascals that lives on the geom and is consumed unmodified by the contact-surface computation. It's still a simplified material model (linear in extent, not linear in strain; not a real continuum-mechanics solve), but it's a constitutive description, not a numerical one. The user-facing question changes from "what values make the simulation stable?" to "what material is this meant to be?" — and that question has answers in physics textbooks rather than in tuning notebooks.

4.2. The equilibrium surface

The central operation in hydroelastic contact is: given two pressure fields that overlap in space, find the surface where they balance.

Why does that surface matter? Picture the simplest case: a soft sphere pressed against a soft block, with the sphere a bit stiffer (larger E) than the block. As the sphere pushes in, both materials "want" to occupy the same region, and the laws of physics pick a compromise: the sphere indents the block more than the block indents the sphere, with the boundary between them landing wherever the block's outward push and the sphere's outward push are equal. That boundary is the equilibrium surface. By Newton's third law, the contact force at any point on it is determined by the local pressure (which is the same on both sides because we're at equilibrium), and the integral of pressure times normal over the whole surface gives the net wrench on each body.

Hydroelastic computes this surface exactly, in closed form, by exploiting the fact that pressure fields are linear within each tetrahedron. Linear fields equate on a plane — that's just algebra — so for any pair of overlapping tetrahedra, there's an analytical equilibrium plane:

geometry/proximity/field_intersection.h:54

// Compute the equilibrium plane of two tetrahedra,
// one from each volume mesh field. The plane is where
//   field0(x) = field1(x)
// with normal n_hat = (grad f0 - grad f1) / ||grad f0 - grad f1||.
bool CalcEquilibriumPlane(int element0, int element1,
                         const Field& f0, const Field& f1,
                         Plane<T>* plane);

Two observations. The plane's normal direction is the difference of the two fields' gradients (normalized). If the two fields had the same gradient, the equilibrium plane would have no preferred direction — which makes sense, because the two materials would be indistinguishable in that region. The position of the plane is wherever the two scalar fields take the same value, which is fully determined by the intercepts. So the equilibrium plane is a deterministic function of the local geometry and material constants, with no iteration needed.

Now the question is: where does this plane intersect the actual geometry? Both tets are bounded; the equilibrium plane is infinite. We want only the portion of the plane that's inside both tets:

geometry/proximity/field_intersection.h:90

// Compute the polygon = (tet0 ∩ tet1 ∩ equilibrium-plane).
bool IntersectTetrahedra(int e0, int e1,
                        const Mesh& m0, const Mesh& m1,
                        std::vector<Vector3<T>>* polygon);

This is a 3D polygon-clipping problem. Each tet is the intersection of 4 half-spaces (its 4 faces), so two tets together define a convex region as the intersection of 8 half-spaces. The equilibrium plane adds one more constraint. Drake clips a starting polygon (the equilibrium plane, large enough to contain both tets) against each of the 8 half-spaces in turn, in classical Sutherland-Hodgman style:

geometry/proximity/mesh_intersection.h:79

void ClipPolygonByHalfSpace(...);

After 8 clipping passes, what survives is a convex polygon — the slice of the equilibrium plane that's inside the tet-tet overlap region. That polygon, with its vertices, is one "face" of the contact surface. Its area times the local pressure gives the local contribution to the contact force.

Now scale this up. A real geom has hundreds or thousands of tetrahedra. In principle every tet of geom A could overlap with every tet of geom B — quadratic blowup — but in practice only a tiny fraction overlap (the contact region is small compared to the overall volume). Drake culls the candidate pairs with a bounding-volume hierarchy on each tet mesh:

geometry/proximity/bvh.h:32

template <class MeshType>
class Bvh {
  // Hierarchical bounding boxes; recursively descend
  // into both trees in tandem to find candidate pairs.
};

This is essentially "broadphase, but inside the narrowphase": bounding-volume tree intersection in O(log n) per query, the same trick MuJoCo and PhysX use at the geom level, just applied here at the tetrahedron level inside one geom pair. Without it, hydroelastic would be unusably slow on anything but small meshes.

The driver that orchestrates the per-tet computation is:

geometry/proximity/field_intersection.h:188

// Iterate over candidate tet pairs (BVH-accelerated),
// compute equilibrium plane and polygon for each,
// accumulate into surface_M with pressure field e_M on it.
void IntersectFields(const Field0& f0_M, const Field1& f1_N,
                     PolygonSurfaceMesh<T>* surface_M,
                     PolygonSurfaceMeshFieldLinear<T, T>* e_M);

Each surviving tet pair contributes one polygon to the output mesh (surface_M) and one piece of the pressure field (e_M) defined linearly on that polygon. Stitching the polygons together is automatic — they share edges where their source tets share faces — and produces a discrete contact surface that's piecewise planar, with a piecewise-linear pressure field defined on it.

And the soft-soft entry point that contact code actually calls:

geometry/proximity/field_intersection.h:348

ContactSurface<T> ComputeContactSurfaceFromCompliantVolumes(...);

Rigid-soft contact is a separate dispatch that intersects a surface mesh against a tet-mesh field (hydroelastic_calculator.h:22). The output type is the same. Mathematically it's the limit of soft-soft as one geom's modulus goes to infinity: the equilibrium surface collapses onto the rigid geom's surface, which is exactly what the rigid-soft path computes directly. Skipping a step that produces a known answer is faster than computing it through the limit.

Why polygons (instead of triangulating)? The output mesh stores polygons of arbitrary vertex count (up to 8, since you can't get more than 8 vertices from clipping a plane by 8 half-spaces). Triangulating them would multiply the number of mesh elements without adding information — the pressure field is linear over each polygon already, so each polygon only needs one quadrature sample to integrate exactly. Drake keeps the polygons whole. There's also a triangle-mesh output mode if downstream code requires it, hence the std::variant in the result type (we'll see this in §4.3).

4.3. The output: ContactSurface

Where MuJoCo gives you mjContact (a per-point geometric record) and PhysX gives you PxContactPatch (a small bundle of points sharing a normal), Drake gives you a mesh:

geometry/query_results/contact_surface.h:160

class ContactSurface {
  // The contact-surface mesh itself: either triangle or polygon.
  std::variant<std::unique_ptr<TriangleSurfaceMesh<T>>,
               std::unique_ptr<PolygonSurfaceMesh<T>>> mesh_W_;

  // Pressure field e(x) defined on the mesh.
  // e_MN_W: pressure of M evaluated at points on the mesh,
  // in world frame.
  ...field_;

  // Per-face gradients of the two pressure fields, ∇e_M and ∇e_N,
  // kept piecewise constant per face.
  ...grad_eM_W_, grad_eN_W_;

  // Mesh normal points "out of N and into M".
};

Walking through the fields:

  • The mesh itself is either a polygon mesh (the raw output of clipping, exact and compact) or a triangle mesh (triangulated for downstream code that wants triangles, e.g. visualization). The std::variant means consumers pick which form they want at construction.
  • The pressure field on the mesh, denoted eMN, is the per-vertex pressure value. For polygon outputs this is a piecewise-linear field; for triangle outputs it's piecewise-linear in barycentric coordinates over each triangle. Either way the mesh and the field together are enough to evaluate the local pressure anywhere on the surface.
  • The two field gradients, ∇eM and ∇eN, are stored as piecewise-constant per face. These exist not because the force computation needs them — pressure on the surface is enough — but because downstream friction models need to know how the local field direction relates to the actual material gradient. They also let consumers compute differential quantities (e.g. for derivatives of the contact force with respect to body pose) without re-doing the surface intersection.
  • The normal convention — mesh normal points "out of N and into M" — matters because it's how the force on each body gets the right sign. By convention, M is the "front" of the contact pair and N is the "back"; the per-vertex pressure eMN is the pressure of M at this point, and the normal points into M, so pressure×area×normal gives a force pushing M away from N. Newton's third law handles N's force as the negative.

This struct is the heart of the hydroelastic philosophy. The contact between two bodies is not a list of points decorated with tuning knobs; it's a field on a discrete surface, and the force will come from integrating that field. There's no solref here, no maxImpulse, no combine-mode-selected friction coefficient sitting on the contact itself. Material parameters are upstream (on the geoms); the surface is just a geometric+field object that downstream code integrates over.

5. From contact surface to spatial force

Once we have a contact surface (or a point pair, in the kPoint case), it has to become a force on each body. For point contact this is a single per-pair force at the contact point. For hydroelastic, the force is an integral over the surface:

F = ∫S p(x) · n̂(x) dA

where S is the contact surface, p(x) is the local pressure (from the field on the mesh), and n̂(x) is the surface normal. Drake evaluates this integral by quadrature: sample the integrand at one or more well-chosen points per face, weight by the face area, sum.

Because the pressure field is linear over each polygon face and the normal is constant per face (the face is planar by construction), one quadrature point per face is enough to integrate the normal force exactly. The natural choice is the face centroid, where the linear field's value equals its average, so:

Fface = p(xcentroid) · Aface · n̂face

Sum these face forces over all polygons in the surface, and you have the net force on body M (with the opposite sign on body N by Newton's third law). The torque about the surface centroid comes from the cross-product of each face's centroid-to-surface-centroid offset with the face force; the discrete-update code does this shift directly:

multibody/plant/discrete_update_manager.cc:524

// Per-quadrature-point processing on the contact surface:
//   1. Solver returns local force f_Bq_C at quadrature point Q,
//      decomposed into normal fn and tangential ft components.
//   2. Transform to world: f_Aq_W = -(R_WC * f_Bq_C)
//   3. Shift to spatial force about the surface centroid O:
//      Fq_Ao_W = SpatialForce(...).Shift(p_QO_W)
//   4. Accumulate: F_Ao_W_per_surface[i] += Fq_Ao_W

Notice the integrand also has a tangential component (ft), not just normal. That's friction. At each quadrature point, the contact solver (SAP, §6) returns a 3D force with a normal piece determined by the pressure and a tangential piece determined by the friction model. Friction in hydroelastic isn't a per-pair scalar cone the way it is in MuJoCo and PhysX — it's a per-quadrature-point regularized Coulomb law with three pieces of information at each point:

  1. The normal pressure (which sets the cone radius, μp).
  2. The relative sliding velocity at this point (which sets the friction direction and magnitude inside the cone, smoothed with SAP's regularization).
  3. The Hunt-Crossley dissipation coefficient (which adds a normal-direction velocity-proportional damping force at each quadrature point, distinct from SAP's solver-side regularization). This is the user's dissipation value from AddContactMaterial.

This means friction in Drake genuinely varies across the contact surface: a sliding tire has high friction at the rear of the contact patch (where the rubber is being pushed back) and low friction at the front (where it's about to release), and hydroelastic captures this naturally because every face has its own velocity and pressure. MuJoCo and PhysX, with one cone per contact point and 4–5 contact points per pair, can only approximate this at much coarser resolution.

After accumulation, the per-surface result is packaged:

multibody/plant/hydroelastic_contact_info.h:35

class HydroelasticContactInfo {
  const ContactSurface<T>* contact_surface_;
  SpatialForce<T> F_Ac_W_;   // net spatial force at centroid
};

One HydroelasticContactInfo per pair of contacting geoms. The pointer to the contact surface is kept around for visualization and downstream queries (e.g., users sometimes want to inspect the pressure distribution, not just the net force); the SpatialForce is what the dynamics actually consume.

6. The constraint solver: SAP

Once Drake has the geometric and material data assembled into a set of contact constraints, it hands them to SAP — the Semi-Analytic Primal solver. SAP is convex like MuJoCo's solver, and similarly formulates contact as a constrained optimization. The differences are in what variable space the solve happens in, what gets regularized, and how friction is modeled.

6.1. What SAP is, in plain terms

Three words, "Semi-Analytic Primal":

  • Primal: the optimization variable is the velocity of each body at the next timestep. "Primal" is in contrast to dual formulations like LCPs or QPs in impulse space, where the variable is the contact impulse and you recover velocity afterward. SAP picks velocity as primary because the cost function (kinetic energy plus contact penalty) is more naturally expressed in velocity than in impulse, and because the velocity space's structure matches the regularized friction formulation cleanly.
  • Semi-Analytic: per-constraint subproblems are solved in closed form. SAP doesn't iterate inside each contact; given the candidate velocity, the contact impulse and friction direction at that velocity have an analytic solution, which the outer Newton loop uses as gradient/Hessian information. This is the trick that makes the convex solve fast.
  • Convex: the cost function is strictly convex, so the global minimum is unique and Newton-style methods are guaranteed to converge.

multibody/contact_solvers/sap/sap_solver.h:17

class SapSolver {
  // Solves the discrete contact problem as a smooth convex
  // optimization in the primal velocity variables.
  // Uses a semi-smooth Newton method with line search.

  // Parameters: rel_tolerance, abs_tolerance, line_search_type,
  // cost_criterion (used as fallback if optimality cannot be
  // reached due to ill-conditioning).
};

Mechanically: SAP iteratively updates the candidate post-step velocity by solving a Newton system at each iteration, with a line search to ensure the cost decreases. The cost combines kinetic energy (deviation from the unconstrained free-flight velocity, weighted by mass) and a contact penalty term per constraint. The contact penalty grows quadratically when the constraint is violated and is zero when it's satisfied, which makes it convex and differentiable. Iterations stop when both gradient norm and step size are below tolerance. The cost_criterion fallback handles ill-conditioned cases (very stiff contact, near- singular Jacobians) where the Newton step can't fully resolve the problem; in those cases SAP returns the lowest-cost iterate it found.

6.2. Friction: a smooth cone, not an LCP

Classical rigid-body contact solvers (Stewart-Trinkle, the original Bullet, parts of PhysX's PGS) formulate friction as a linear complementarity problem (LCP): the tangential impulse is "either inside the cone with zero sliding, or on the cone boundary with sliding in a specific direction" — a binary case split per contact, embedded in the math as complementarity conditions. LCPs are non-smooth, NP-hard in general, and have many degenerate cases.

SAP avoids this entirely by regularizing the cone:

multibody/contact_solvers/sap/sap_friction_cone_constraint.h:28

class SapFrictionConeConstraint {
  // Friction impulse gamma_t obeys:
  //   ||gamma_t|| <= mu * gamma_n   (kStiction)
  //   ||gamma_t||  =  mu * gamma_n   (sliding)
  // with regularization parameters Rt (tangent) and Rn (normal)
  // adding small compliance to make the cone smooth.

  double Rt_, Rn_;  // regularization (small, positive)
};

Concretely: instead of "no sliding inside the cone," SAP says "very small sliding inside the cone, proportional to Rt." Instead of "no normal compression below zero," SAP says "very small compression possible, proportional to Rn." The cone becomes a smooth function and the whole problem becomes twice-differentiable, making Newton's method directly applicable.

The regularization parameters are small enough that physical predictions match real Coulomb friction within solver tolerance. Drake picks them automatically based on the contact stiffness and timestep; users don't typically tune Rt or Rn directly.

6.3. Comparison with MuJoCo's solver

Both MuJoCo's Newton solver (Part 2, §7) and SAP solve a convex optimization with primal Newton iterations. The key differences:

  • Variable space. MuJoCo solves in acceleration space (the dual is in contact-force space). SAP solves in velocity space directly. For a discrete-time integrator both are equivalent up to a timestep multiplication; the choice affects the Hessian's conditioning.
  • What's regularized. MuJoCo regularizes the constraint (penetration is allowed in proportion to solref/solimp). SAP regularizes the friction cone (tiny stiction creep) and only mildly the normal direction (tiny compression allowed). MuJoCo's regularization is shaped like a stiffness; SAP's is shaped like a smoothing constant.
  • Where physical parameters enter. In MuJoCo, the "stiffness" and "damping" of contact come from solref/solimp — solver-side knobs. In SAP, contact stiffness comes from the hydroelastic pressure field upstream, and SAP's regularization is just enough smoothing to make the math well-behaved.

The two important consequences of SAP's design:

  • No LCP code path. Friction is regularized, the friction cone is smooth, and the whole problem is solvable with a primal Newton method — the same family of solver as MuJoCo's Newton, but with friction handled differently.
  • Dissipation is material-rooted, not solver-rooted. The damping in a hydroelastic contact comes from the Hunt-Crossley dissipation parameter the user set on the geom via AddContactMaterial (proximity_properties.h:114). SAP's regularization adds a small numerical compliance to make the cone smooth, but the physical dissipation is upstream — specified in physical units (1/(m/s) Hunt-Crossley dissipation), not in solver parameters.

Compare with MuJoCo, where the K·r−B·v reference acceleration's B term is a tunable solver parameter that doubles as a damping model, and PhysX, where damping is a material-level coefficient but only takes effect via the negative-restitution back door. SAP separates the two cleanly: material physics goes in through the contact surface; numerical conditioning goes in through SAP's regularization.

7. Time stepping

MultibodyPlant by default runs in discrete-time mode, advancing state by a fixed timestep on each call. Drake supports continuous-time integration through the System framework's variable-step ODE solvers, but in practice the discrete plant is what almost everyone uses for contact-rich simulation, because it's the mode SAP was designed for.

Why discrete? Because SAP's convex formulation is implicitly an integrator: when you solve for the post-step velocity that minimizes "kinetic-energy distance from free flight" plus contact penalty, the result is the velocity-update of an implicit-in-velocity scheme. Solve once, advance by Δt, repeat. There's no separate ODE integrator wrapped around the contact solve; the contact solve is the integrator step. This is similar in spirit to MuJoCo's implicit-in-velocity reference acceleration (Part 2, §6), achieved by a different mechanism — SAP through convex formulation, MuJoCo through co-solving constraint forces and accelerations.

The consequence: stiff contact stays stable at large timesteps. A 1 GPa hydroelastic geom — harder than steel — in Drake doesn't need a sub-millisecond timestep the way an explicit integrator would. The convex formulation absorbs the stiffness into its conditioning rather than into the timestep budget.

The contact pipeline is driven once per discrete update through this entry point:

multibody/plant/discrete_update_manager.h:150

virtual void CalcContactSolverResults(...) const = 0;

Per step, the manager:

  1. Queries SceneGraph for point pairs and/or contact surfaces, depending on the contact model. The query returns "everything currently in contact" — no reuse from the previous step; this is recomputed.
  2. Builds discrete contact pairs with the constraint Jacobians (the matrices that map body velocities to contact-point velocities) and the combined friction coefficients.
  3. Hands those to SAP, which solves for post-step velocities and contact impulses.
  4. Integrates impulses into spatial forces: for point-contact pairs, this is just a force at the contact point; for hydroelastic, it's the quadrature loop from §5 (pressure×area×normal at each face, plus friction at each face).
  5. Advances state: applies the post-step velocity, integrates positions over Δt.

Note Drake recomputes contact from scratch each step. Unlike PhysX, there's no persistent contact manifold reused across frames. The justification is that hydroelastic's surface computation is itself a continuous function of body pose — no warm-start needed, because the discontinuities the persistent manifold protects against in PhysX (contact birth/death) don't exist for hydroelastic in the first place.

The discrete pair carries minimal data:

multibody/plant/discrete_contact_pair.h:70

struct DiscreteContactPair {
  GeometryId id_A, id_B;
  int surface_index;       // which contact surface (if hydro)
  RotationMatrix<T> R_WC;  // contact frame rotation
  Vector3<T> p_WC;         // contact point / centroid (world)
  T friction;              // combined friction coefficient
};

Notice this struct is the closest analog Drake has to MuJoCo's mjContact — a per-pair record passed to the solver. But it carries pointers into the contact surface (surface_index), so the integration step has access to the full mesh and field, not just a per-point summary. Most of the per-contact data is held by the ContactSurface the index points back to; this struct is mostly bookkeeping.

8. The architectural commitments, in code

Three things distinguish Drake's contact pipeline at the source-tree level. None of them are hidden — they're visible from a glance at the file layout:

  1. Geometry and dynamics are separate components. geometry/proximity_engine.h owns shapes and queries. multibody/plant/multibody_plant.cc owns inertia and time-stepping. They communicate over a System port. This is why Drake can switch between point and hydroelastic without touching the dynamics, and why deformable-body support could be added without rewriting the rigid-body solver.
  2. Material parameters live on geoms, not on solvers. AddCompliantHydroelasticProperties takes a Young's modulus. AddContactMaterial takes a friction coefficient and a Hunt-Crossley dissipation. Nowhere does the user tune a regularization parameter shaped like a stiffness; the regularization (in SAP) is a numerical detail with sensible defaults, not a knob.
  3. Contact has spatial extent by construction. ContactSurface is a mesh with a field on it. The rest of the pipeline integrates that field. There's no contact-reduction step, no "pick four points," no manifold cache — the surface itself is the contact, and its fineness is set by the user's resolution_hint when the geom was meshed.

This is what people mean when they call hydroelastic "contact-as-constitutive-law": the force at a contact point is prescribed by state (p(x) = E·ε(x)), the surface is an integral domain, and the solver's job is to enforce Newton's laws and friction — not to invent the contact force.

9. Recap: a single timestep, narrated

  1. Geometry query (SceneGraph): for each candidate pair, either compute a PenetrationAsPointPair (kPoint) or a ContactSurface with pressure field (kHydroelastic), or both with fallback.
  2. Pressure field intersection (hydroelastic only): for each overlapping tet pair, solve the linear equilibrium plane, clip the two tets against it, accumulate polygons into a surface mesh.
  3. DiscreteContactPair construction: build the Jacobians, friction coefficients, and references to surface data the solver needs.
  4. SAP solve: smooth convex optimization in primal velocity, with regularized friction cones.
  5. Spatial force integration: for each hydroelastic surface, quadrature over the mesh turns pressure×normal×area into a wrench at the surface centroid.
  6. Discrete update: apply impulses, advance state.

The core difference from Parts 2 and 3 is at step 1. MuJoCo and PhysX both produce a list of contacts and tuning knobs; Drake produces a contact surface and a material-rooted pressure field. Everything downstream follows from that.

10. A short reading list

  • Elandt, Drumwright, Sherman, Ruppel, "A Pressure Field Model for Fast, Robust Approximation of Net Contact Force and Moment Between Nominally Rigid Objects" (IROS 2019). The paper that introduces hydroelastic contact. Short and readable. Read this before the source.
  • Drake's hydroelastic user guide (doc/_pages/hydroelastic_user_guide.md in the Drake repo). Practical documentation with diagrams of equilibrium surfaces, and the most useful guide to the modulus and resolution-hint values to actually use.
  • Castro, Permenter, Han, "An Unconstrained Convex Formulation of Compliant Contact" (T-RO 2023). The paper behind SAP. Worth reading after the hydroelastic paper, because the formulation is what makes the integrator work well at large timesteps.
  • geometry/proximity/field_intersection.h, geometry/query_results/contact_surface.h, multibody/contact_solvers/sap/sap_solver.h. The three files together cover most of the pipeline above.
